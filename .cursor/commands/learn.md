# /learn — 提取可复用模式

分析当前会话，提取能在未来项目中复用的经验，并在用户确认后保存为项目知识。

## 触发时机

在会话中解决了非平凡问题，且结论不依赖一次性环境、临时数据或个人偏好时运行。

## 可提取类型

1. **错误解决模式**：症状、根因、修复与适用前提。
2. **调试技巧**：有效的诊断顺序、工具组合和排除路径。
3. **工程模式**：接口设计、并发、错误处理、测试隔离、性能或安全实践。
4. **项目约定**：经验证的架构决策、集成方式、构建与测试惯例。
5. **外部限制的处理方式**：第三方库、平台或版本限制及可验证的解决方案。

不要把语言、框架或当前项目的偶然细节包装成“通用规律”。技术栈特定经验必须明确其适用范围；一次性问题和简单拼写/语法修复不应保存。

## 输出格式

建议在 `.cursor/skills/learned/` 创建 kebab-case 文件：

```markdown
# [描述性模式名称]

**提取日期:** YYYY-MM-DD
**适用场景:** [何时使用，以及语言/框架/项目等前提]

## 问题

[具体症状与影响]

## 根因

[已验证的根本原因]

## 解决方案

[可复用步骤和约束]

## 示例

```text
[用当前技术栈的最小示例；没有必要时省略]
```

## 验证方式

[如何确认模式适用且修复有效]

## 何时使用

[明确触发条件]

## 不适用场景

[已知边界与反例]
```

## 执行流程

1. 审查会话并识别候选模式。
2. 验证它不是一次性结论，并写明适用边界。
3. 起草一个聚焦的 Markdown 文件；一个文件只记录一个模式。
4. 向用户展示草案并请求确认。
5. 获确认后保存到 `.cursor/skills/learned/`。

## Hook 自动注入

项目的 session-start hook 会枚举 `.cursor/skills/learned/` 中的 Markdown 文件并注入索引。新增文件无需维护额外索引。
